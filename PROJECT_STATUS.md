# 📜 VibeBass 프로젝트 개발 현황 및 로드맵 (PROJECT_STATUS)

본 문서는 VibeBass(악보-유튜브 연동 연주 플랫폼)의 지금까지의 구현 성과와 트러블슈팅 이력, 그리고 실서비스 출시를 위해 필요한 향후 개발 과제(로드맵)를 정리한 프로젝트 공식 명세서입니다.

---

## 📅 프로젝트 개요
* **프로젝트명**: VibeBass (바이브베이스)
* **기술 스택**: Kotlin Multiplatform (WasmJs Compose), Kotlin Spring Boot 3.3.4, Spring Data JPA, PostgreSQL (JSONB), YouTube Data API v3

---

## ✅ 1. 지금까지 구현 완료된 기능 (현재 상태: MVP)

### 🖥️ 프론트엔드 (Compose Multiplatform / WasmJs)
*   **지능형 PDF 악보 뷰어**: 로컬 PDF 악보를 업로드하면 1페이지 텍스트 데이터를 분석하여 자동으로 곡의 제목과 가수를 추출합니다.
*   **유튜브 플레이어 양방향 브릿지**: 추출된 노래 정보를 바탕으로 백엔드를 통해 YouTube 비디오 ID를 자동 검색하여 화면에 바인딩합니다. 재생 시간과 상태(재생/일시정지)가 실시간으로 코틀린 UI 단에 바인딩됩니다.
*   **Spacebar 실시간 싱크 수집**: 싱크 활성화 모드에서 음악 재생에 맞춰 스페이스바를 누르면, 현재 재생 시간과 악보의 실시간 스크롤 픽셀 위치가 1:1 앵커 포인트로 매핑되어 리스트에 적재됩니다.
*   **선형 보간(Linear Interpolation) 감쇠 스크롤**: 수집 및 로드된 앵커 포인트를 바탕으로 곡 재생 흐름에 맞춰 악보가 부드럽게 감쇠하며 자동 스크롤됩니다.
*   **60fps 실시간 플레이헤드(재생 가이드선)**: Compose의 연산 부하를 피하기 위해 브라우저의 DOM 단에서 가상의 빨간색 재생선을 띄우고 스크롤 이동 시 CSS Transition(`0.15s ease-out`)과 네온 효과로 부드럽게 이동하게 만들어 시각적 연주 가독성을 극대화했습니다.
*   **노래 목록 대시보드 (CRUD)**: 좌측 패널에 가로 50:50 분할 패널을 구축해 실시간 수집 앵커와 DB에 저장된 곡 목록을 대시보드화했으며, 목록 클릭 시 저장된 유튜브 비디오, PDF 파일 이름 정보, 앵커 정보가 원클릭으로 즉시 로드됩니다.

### ⚙️ 백엔드 (Kotlin Spring Boot & JPA)
*   **코틀린 단일화**: 프로젝트 일관성과 널 안전성을 보장하기 위해 기존 Java 백엔드를 Kotlin 스택으로 100% 전향 및 마이그레이션했습니다.
*   **PostgreSQL JSONB 앵커 보관**: Hibernate 6의 `@JdbcTypeCode(SqlTypes.JSON)`를 활용해 자바의 앵커 포인트 객체 리스트를 DB의 JSONB 타입 컬럼과 양방향 자동 직렬화 매핑하였습니다.
*   **논리 삭제(Soft Delete)**: `@SQLDelete` 및 `@Where` 설정을 JPA 엔티티 단에 추가해 데이터 물리 삭제를 방지하고 `deleted_at` 갱신 논리 삭제 처리를 자동화했습니다.
*   **회복탄력적 유튜브 자동 검색 API**: WebClient에 5초 타임아웃을 명시하고 4xx/5xx 예외를 개별 포착하며, 멱등성 읽기 장애 시 최대 3회의 지수 백오프 재시도(Exponential Backoff Retry)를 수행하는 검색 엔진을 탑재했습니다.

### 📦 빌드 파이프라인 자동화 (배포 표준화)
*   **단일 JAR 패키징**: `copyWebDist` Gradle 태스크를 구축하여 웹앱 빌드 산출물(HTML/JS/Wasm/CSS)이 백엔드의 빌드 리소스 경로(`build/resources/main/static`)에 패키징 시점에 자동 병합되도록 엮었습니다.
*   **원스톱 가동**: `java -jar backend-0.0.1-SNAPSHOT.jar` 명령어 한 줄로 프론트엔드 정적 파일 서빙과 백엔드 데이터베이스 통신이 하나의 포트(`8082`)에서 일괄 실행됩니다.

## 🛠️ 해결된 주요 기술 이슈 (Troubleshooting)

1.  **DPR(Device Pixel Ratio) 팽창 버그 교정**:
    *   *이슈*: Retina 디스플레이 등 고해상도 환경에서 픽셀 배율 차이와 포커스 변경에 따른 리컴포지션으로 인해 유튜브 창이 비정상적으로 커지거나 싱크가 어긋남.
    *   *해결*: Kotlin 단의 리액티브 픽셀 연산을 걷어내고 CSS 절대반응형 수식(`calc(40% - 32px)`, `left: 16px`, `top: 64px`) 매핑으로 좌표 정렬을 일원화하여 해결.
2.  **Gradle 9.x / Kotlin 2.4.0 호환성 충돌**:
    *   *이슈*: `io.spring.dependency-management` 플러그인의 구식 API 참조로 인한 Gradle 빌드 차단 및 JVM 컴파일러 타겟 불일치 발생.
    *   *해결*: 플러그인을 제거하고 `spring-boot-dependencies:3.3.4` BOM 플랫폼 임포트 방식으로 개량, JVM 툴체인을 17로 통일하고 스프링 부트를 3.3.4로 격상하여 해결.
3.  **WasmJS 인라인 스크립트(js) 호출 제약**:
    *   *이슈*: `js(...)` 호출부 코드가 클래스 멤버 함수에 포함되어 컴파일 실패 발생.
    *   *해결*: 브릿지 함수(`saveSyncDataJs`, `loadSongsJs`)를 WasmJs actual 파일의 **최상단 탑레벨(Top-level) 독립형 함수**로 완벽하게 격리해 해결.
4.  **WebClient 절대 경로 호스트 누락 예외**:
    *   *이슈*: WebClient가 전체 검색 URL을 파싱할 때 `Host is not specified` 500 에러를 터뜨림.
    *   *해결*: `UriComponentsBuilder`를 활용해 완전한 절대 경로 URI 객체를 구성해 직접 바인딩하여 정상 호출 성공.

## 🚀 2. 앞으로 구현해야 할 기능 (향후 로드맵)

정식 릴리즈 및 서비스 배포를 위해 고도화가 필요한 핵심 과제들입니다.

### 📅 1순위 과제: PDF 파일 업로드 및 클라우드 스토리지 보관 기능
*   **현재 한계**: 사용자가 로컬에서 PDF를 선택할 때 브라우저의 임시 Object URL로만 띄우기 때문에, 노래 목록을 통해 불러왔을 때 다른 컴퓨터나 브라우저에서는 실제 악보 이미지가 화면에 로드되지 않습니다.
*   **구현 방향**:
    1.  스프링 백엔드 서버에 `MultipartFile` 업로드를 지원하는 파일 업로드 API(`POST /api/songs/upload`)를 신규 개설합니다.
    2.  업로드된 PDF 파일을 서버의 지정 디렉토리 혹은 AWS S3 등의 클라우드 스토리지에 영구 저장합니다.
    3.  `Song` 엔티티에 `pdfFileUrl` 속성을 추가해 저장하고, 노래 데이터를 불러올 때 저장된 웹 주소(URL)를 통해 악보 뷰어가 파일을 완벽하게 로드하도록 수정합니다.

### 📅 2순위 과제: 저장 전 곡명/가수 정보 수동 수정 UI 지원
*   **현재 한계**: PDF 텍스트 추출기에만 전적으로 의존해 간혹 노래 제목에 도메인 텍스트가 섞이거나 가수가 잘못 파싱될 우려가 있습니다.
*   **구현 방향**:
    *   자동 추출 결과를 화면 상단 텍스트 필드(Input Form)에 먼저 기본값으로 보여주고, 사용자가 마음에 들지 않으면 키보드로 **곡명과 가수 이름을 직접 수정 및 보완해서 백엔드에 저장**할 수 있게끔 입력 폼 UI를 추가합니다.

### 📅 3순위 과제: 사용자 계정 및 개인 서재 목록 분리 (Security)
*   **현재 한계**: 로그인 기능이 없어 누구나 저장한 노래 목록이 DB에 무방비로 공용 노출됩니다.
*   **구현 방향**:
    *   Spring Security 및 JWT(JSON Web Token) 또는 OAuth2 소셜 로그인을 탑재하여 로그인한 사용자 정보(`userId`)를 식별하고, 오직 사용자가 직접 저장한 노래 목록만 조회/저장되도록 데이터베이스 쿼리를 계정별로 필터링 격리합니다.

### 📅 4순위 과제: 악보 위 실시간 재생 바 및 마디 하이라이팅 고도화
*   **현재 한계**: 현재는 실시간 가로선(Playhead) 가이드 바만 제공됩니다.
*   **구현 방향**:
    *   사용자가 스페이스바를 칠 때 스크롤 픽셀 정보뿐 아니라 악보 내에서 마디가 위치한 사각형 박스 영역(Bounding Box)을 연동 등록하게 만들어, 노래가 연주될 때 현재 치고 있는 **해당 마디 또는 가사 영역 전체가 노란색 반투명 박스로 번쩍이며 실시간 하이라이팅**되는 고도의 비주얼 가이드 레이어를 구축합니다.
